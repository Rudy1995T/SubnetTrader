# SubnetTrader Safety & Guardrails — Specification

## Overview

Harden the first-run experience and control panel so that new users cannot accidentally
trade live without understanding the risks, and operators can see at a glance whether
every dependency (RPC, Taostats, Telegram, wallet) is healthy.

**Goal:** a user who has never run the bot can complete setup without risking funds, and
any operator — new or experienced — can diagnose connection or config issues from a single
dashboard page without touching the terminal.

## Part 1 — First-Run Safety Defaults

### 1.1 Default Configuration (already in place)

`EMA_DRY_RUN` defaults to `true` in `app/config.py`. No change needed — this section
documents the invariant and adds enforcement.

**Invariant:** a fresh install (no `.env` file, or `.env` generated by the setup wizard
with a "Conservative" preset) MUST start with `EMA_DRY_RUN=true`. The setup wizard
already defaults the Conservative preset to `true`; the Moderate and Aggressive presets
set it to `false` but only after the user explicitly selects them.

---

### 1.2 Setup Wizard — Live-Trading Safety Gate

When the user toggles `EMA_DRY_RUN` to **OFF** (live trading) in Step 4 (Trading Mode)
of the setup wizard, or selects the Moderate / Aggressive preset, the wizard shows an
interstitial confirmation before the toggle takes effect.

#### Trigger

Any action that would set `EMA_DRY_RUN = false`:
- Clicking the **Moderate** or **Aggressive** preset card.
- Manually toggling the "Paper Trading Mode" switch to OFF.

#### Safety Gate UI

When triggered, a modal overlay appears **inside** the Step 4 card (not a browser dialog):


#### Pre-flight Checks

The modal calls `GET /api/health/services` (new endpoint, see Part 2) and
`POST /api/config/wallet/validate` to populate the checklist. Each check is one of:

| Icon | Meaning | Colour |
|------|---------|--------|
| ✓ | Pass | `text-emerald-400` |
| ✗ | Fail (blocking) | `text-red-400` |
| — | Skipped / optional | `text-gray-500` |

**Blocking checks** (must pass before the "Enable Live Trading" button is clickable):

1. **Wallet configured** — `BT_WALLET_NAME` and `BT_WALLET_HOTKEY` are set (from
   wizard form state).
2. **Wallet unlockable** — `POST /api/config/wallet/validate` returns
   `checks.coldkey_unlockable: true`.
3. **Balance sufficient** — `checks.balance_tao >= EMA_POT_TAO` (from the pot size
   entered on the same page).

**Non-blocking checks** (informational, do not prevent enabling):

4. **RPC connected** — `GET /api/health/services` returns `rpc.ok: true`. If false,
   show warning but allow proceeding (the bot will fall back to subtensor).
5. **Taostats reachable** — `services.taostats.ok: true`. If false, show warning.
6. **Telegram configured** — `TELEGRAM_BOT_TOKEN` and `TELEGRAM_CHAT_ID` are set. If
   not, show "not configured (optional)".

#### Confirmation Checkbox

The user must check "I understand that live trading uses real funds" before the
"Enable Live Trading" button becomes clickable. This is a simple `useState<boolean>`
guard — no backend persistence.

#### Behaviour on Confirm

1. The modal closes.
2. The "Paper Trading Mode" toggle flips to OFF.
3. The red live-trading warning bar appears (existing behaviour from the onboarding spec).
4. The preset label updates to reflect the selected state.

#### Behaviour on Cancel

Modal closes. The toggle stays ON (paper mode). Preset selection resets to Conservative
if the user had clicked Moderate/Aggressive.

### 1.4 Backend: Go-Live Endpoint

#### `POST /api/config/go-live`

A convenience endpoint that the safety gate modal can call to perform all pre-flight
checks in a single request. This avoids the frontend needing to orchestrate multiple
API calls.

**Request body:**

```json
{
  "wallet_name": "trader_wallet",
  "hotkey": "trader_hotkey",
  "wallet_path": "~/.bittensor/wallets",
  "password": "...",
  "pot_tao": 10.0
}
```

**Response:**

```json
{
  "checks": {
    "wallet_configured": { "ok": true, "detail": "trader_wallet / trader_hotkey" },
    "wallet_unlockable": { "ok": true, "detail": "5CJVx...36Sf" },
    "balance_sufficient": { "ok": true, "detail": "12.34 TAO (pot: 10.0 TAO)" },
    "rpc_connected": { "ok": true, "detail": "FlameWire healthy" },
    "taostats_reachable": { "ok": false, "detail": "HTTP 401 — check API key" },
    "telegram_configured": { "ok": false, "detail": "Not configured (optional)", "optional": true }
  },
  "can_go_live": true
}
```

**Logic:**

- `can_go_live` is `true` only when all non-optional checks pass (`wallet_configured`,
  `wallet_unlockable`, `balance_sufficient`).
- Each check has `ok` (bool) and `detail` (human-readable string).
- Optional checks have `"optional": true` so the frontend can render them differently.

**Implementation — `config_api.py`:**

```python
@router.post("/go-live")
async def go_live_preflight(body: dict):
    wallet_name = body.get("wallet_name", "").strip()
    hotkey_name = body.get("hotkey", "").strip()
    wallet_path = body.get("wallet_path", "~/.bittensor/wallets").strip()
    password = body.get("password", "")
    pot_tao = float(body.get("pot_tao", 10.0))

    checks = {}

    # 1. Wallet configured
    if wallet_name and hotkey_name:
        checks["wallet_configured"] = {
            "ok": True,
            "detail": f"{wallet_name} / {hotkey_name}",
        }
    else:
        checks["wallet_configured"] = {
            "ok": False,
            "detail": "Wallet name and hotkey are required",
        }

    # 2. Wallet unlockable + 3. Balance sufficient
    try:
        import bittensor as bt

        expanded = os.path.expanduser(wallet_path)
        wallet = bt.Wallet(name=wallet_name, hotkey=hotkey_name, path=expanded)

        # Try unlock
        try:
            if password:
                wallet.unlock_coldkey(password=password)
            else:
                _ = wallet.coldkey
            ss58 = wallet.coldkeypub.ss58_address
            checks["wallet_unlockable"] = {"ok": True, "detail": ss58}
        except Exception as exc:
            checks["wallet_unlockable"] = {"ok": False, "detail": str(exc)}
            checks["balance_sufficient"] = {"ok": False, "detail": "Cannot check — wallet locked"}

        # Balance check (only if unlock succeeded)
        if checks.get("wallet_unlockable", {}).get("ok"):
            try:
                sub = bt.Subtensor()
                balance = float(sub.get_balance(ss58))
                sufficient = balance >= pot_tao
                checks["balance_sufficient"] = {
                    "ok": sufficient,
                    "detail": f"{balance:.4f} TAO (pot: {pot_tao} TAO)"
                        + ("" if sufficient else f" — short {pot_tao - balance:.4f} TAO"),
                }
            except Exception:
                checks["balance_sufficient"] = {
                    "ok": False,
                    "detail": "Chain unreachable — cannot verify balance",
                }
    except ImportError:
        checks["wallet_unlockable"] = {"ok": False, "detail": "bittensor SDK not installed"}
        checks["balance_sufficient"] = {"ok": False, "detail": "SDK unavailable"}

    # 4. RPC connected
    try:
        from app.chain.flamewire_rpc import FlameWireRPC

        temp_rpc = FlameWireRPC()
        rpc_ok = await temp_rpc.health_check()
        checks["rpc_connected"] = {
            "ok": rpc_ok,
            "detail": "FlameWire healthy" if rpc_ok else "FlameWire unreachable",
        }
        await temp_rpc.close()
    except Exception as exc:
        checks["rpc_connected"] = {"ok": False, "detail": str(exc)}

    # 5. Taostats reachable
    try:
        import httpx

        taostats_key = settings.TAOSTATS_API_KEY or ""
        headers = {"Authorization": taostats_key} if taostats_key else {}
        async with httpx.AsyncClient(timeout=10) as client:
            r = await client.get(
                "https://api.taostats.io/api/dtao/pool/latest/v1?limit=1",
                headers=headers,
            )
            if r.status_code == 200:
                checks["taostats_reachable"] = {"ok": True, "detail": "Taostats API responding"}
            else:
                checks["taostats_reachable"] = {
                    "ok": False,
                    "detail": f"HTTP {r.status_code} — check API key",
                }
    except Exception as exc:
        checks["taostats_reachable"] = {"ok": False, "detail": str(exc)}

    # 6. Telegram configured
    tg_configured = bool(settings.TELEGRAM_BOT_TOKEN and settings.TELEGRAM_CHAT_ID)
    checks["telegram_configured"] = {
        "ok": tg_configured,
        "detail": "Configured" if tg_configured else "Not configured (optional)",
        "optional": True,
    }

    can_go_live = all(
        checks[k]["ok"]
        for k in ("wallet_configured", "wallet_unlockable", "balance_sufficient")
        if k in checks
    )

    return JSONResponse(content={"checks": checks, "can_go_live": can_go_live})
```

**Timeout:** the entire endpoint should complete within 30 seconds. The longest operation
is the chain balance query (~5–15s on first connection). The frontend shows a spinner
with "Running pre-flight checks..." while waiting.

### 2.1 Backend: Service Health Endpoint

#### `GET /api/health/services`

Returns the health status of every external dependency. Called by the control page on
mount and every 30 seconds.

**Response:**

```json
{
  "timestamp": "2026-03-20T14:30:00Z",
  "services": {
    "rpc": {
      "ok": true,
      "name": "FlameWire RPC",
      "detail": "Connected",
      "last_check": "2026-03-20T14:29:45Z",
      "latency_ms": 142
    },
    "taostats": {
      "ok": true,
      "name": "Taostats API",
      "detail": "200 OK (30 req/min)",
      "last_check": "2026-03-20T14:30:00Z",
      "latency_ms": 310
    },
    "telegram": {
      "ok": true,
      "name": "Telegram Bot",
      "detail": "Bot @my_bot responding",
      "last_check": "2026-03-20T14:30:00Z",
      "latency_ms": 95
    },
    "database": {
      "ok": true,
      "name": "SQLite Database",
      "detail": "WAL mode, 42 trades",
      "last_check": "2026-03-20T14:30:00Z"
    },

```

**Implementation — `app/main.py`:**

```python
@app.get("/api/health/services")
async def api_health_services():
    import time as time_mod
    results = {}
    ts = utc_iso()

    # 1. RPC health
    try:
        t0 = time_mod.monotonic()
        rpc_ok = await rpc.health_check() if rpc else False
        latency = int((time_mod.monotonic() - t0) * 1000)
        results["rpc"] = {
            "ok": rpc_ok,
            "name": "FlameWire RPC",
            "detail": "Connected" if rpc_ok else "Unreachable",
            "last_check": ts,
            "latency_ms": latency,
        }
    except Exception as exc:
        results["rpc"] = {
            "ok": False,
            "name": "FlameWire RPC",
            "detail": str(exc),
            "last_check": ts,
        }

    # 2. Taostats
    try:
        import httpx as httpx_mod

        taostats_key = settings.TAOSTATS_API_KEY or ""
        headers = {"Authorization": taostats_key} if taostats_key else {}
        t0 = time_mod.monotonic()
        async with httpx_mod.AsyncClient(timeout=10) as client:
            r = await client.get(
                "https://api.taostats.io/api/dtao/pool/latest/v1?limit=1",
                headers=headers,
            )
        latency = int((time_mod.monotonic() - t0) * 1000)
        results["taostats"] = {
            "ok": r.status_code == 200,
            "name": "Taostats API",
            "detail": "200 OK" if r.status_code == 200
                else f"HTTP {r.status_code}",
            "last_check": ts,
            "latency_ms": latency,
        }
    except Exception as exc:
        results["taostats"] = {
            "ok": False,
            "name": "Taostats API",
            "detail": str(exc),
            "last_check": ts,
        }

    # 3. Telegram
    tg_token = settings.TELEGRAM_BOT_TOKEN
    tg_chat = settings.TELEGRAM_CHAT_ID
    if tg_token and tg_chat:
        try:
            import httpx as httpx_mod

            t0 = time_mod.monotonic()
            async with httpx_mod.AsyncClient(timeout=10) as client:
                r = await client.get(f"https://api.telegram.org/bot{tg_token}/getMe")
            latency = int((time_mod.monotonic() - t0) * 1000)
            if r.status_code == 200:
                bot_info = r.json().get("result", {})
                bot_name = bot_info.get("username", "unknown")
                results["telegram"] = {
                    "ok": True,
                    "name": "Telegram Bot",
                    "detail": f"Bot @{bot_name} responding",
                    "last_check": ts,
                    "latency_ms": latency,
                }
            else:
                results["telegram"] = {
                    "ok": False,
                    "name": "Telegram Bot",
                    "detail": f"HTTP {r.status_code} — invalid token",
                    "last_check": ts,
                }
        except Exception as exc:
            results["telegram"] = {
                "ok": False,
                "name": "Telegram Bot",
                "detail": str(exc),
                "last_check": ts,
            }
    else:
        results["telegram"] = {
            "ok": False,
            "name": "Telegram Bot",
            "detail": "Not configured"
                + (" (missing token)" if not tg_token else "")
                + (" (missing chat ID)" if not tg_chat else ""),
            "last_check": ts,
        }

    # 4. Database
    try:
        if db:
            count = await db.fetchone("SELECT COUNT(*) as n FROM ema_positions")
            trade_count = count["n"] if count else 0
            results["database"] = {
                "ok": True,
                "name": "SQLite Database",
                "detail": f"WAL mode, {trade_count} trades",
                "last_check": ts,
            }
        else:
            results["database"] = {
                "ok": False,
                "name": "SQLite Database",
                "detail": "Not initialized",
                "last_check": ts,
            }
    except Exception as exc:
        results["database"] = {
            "ok": False,
            "name": "SQLite Database",
            "detail": str(exc),
            "last_check": ts,
        }

    # 5. Wallet
    try:
        if executor:
            balance = await executor.get_tao_balance()
            pot = settings.EMA_POT_TAO
            can_trade = balance >= pot and not settings.EMA_DRY_RUN
            results["wallet"] = {
                "ok": True,
                "name": "Wallet",
                "detail": f"{settings.BT_WALLET_NAME} — {balance:.4f} TAO",
                "can_trade": can_trade,
                "balance_tao": round(balance, 4),
                "pot_tao": pot,
                "last_check": ts,
            }
        else:
            results["wallet"] = {
                "ok": False,
                "name": "Wallet",
                "detail": "Executor not initialized",
                "last_check": ts,
            }
    except Exception as exc:
        results["wallet"] = {
            "ok": False,
            "name": "Wallet",
            "detail": f"Balance check failed: {exc}",
            "last_check": ts,
        }

    return JSONResponse(content={"timestamp": ts, "services": results})
```

**Performance notes:**
- The RPC, Taostats, and Telegram checks make outbound HTTP calls. Total worst-case
  latency: ~30s (3 × 10s timeouts). In practice, healthy services respond in <500ms.
- The frontend should show a loading skeleton for the health section while waiting.
- Results are NOT cached server-side — each call is fresh. The frontend controls polling
  frequency (30s default).

#### Status Indicators

Each service row has three elements:

1. **Status dot** (●) — colour-coded:
   - `text-emerald-400` — healthy (`ok: true`)
   - `text-red-400` — unhealthy (`ok: false`, and service is required)
   - `text-amber-400` — degraded or optional service not configured
   - `text-gray-500` — loading / unknown

2. **Service name** — fixed label (FlameWire RPC, Taostats API, etc.)

3. **Detail text** — from the `detail` field in the API response. Truncated to 40 chars
   with ellipsis if longer.

4. **Latency badge** (optional) — shown when `latency_ms` is present:
   - `text-emerald-400` if < 500ms
   - `text-amber-400` if 500ms–2000ms
   - `text-red-400` if > 2000ms

#### Wallet Row — Enhanced

The wallet row shows additional detail when `ok: true`:

```
● Wallet    trader_wallet — 12.34 TAO    ✓ Can trade
```

or when balance is insufficient:

```
● Wallet    trader_wallet — 2.50 TAO     ⚠ Pot requires 10 TAO
```

or in dry-run mode:

```
● Wallet    trader_wallet — 12.34 TAO    (dry run — no trades)
```

#### Error State Display

When a service is unhealthy, clicking the row expands a detail section:

```
● FlameWire RPC     Unreachable                          ✗
  └─ Connection refused at ws://flamewire.io:443
     Check FLAMEWIRE_API_KEY in Settings.
```

The expanded detail includes:
- The full `detail` string (not truncated).
- A contextual help message based on the service (see Error Messages table below).
- A link to the relevant `/settings` section (if applicable).

#### Error Messages

| Service | Error condition | Help message |
|---------|----------------|-------------|
| RPC | Unreachable | "Check your internet connection and FLAMEWIRE_API_KEY. The bot will fall back to the public subtensor endpoint." |
| RPC | Timeout | "FlameWire is slow to respond. Trading will continue but may be delayed." |
| Taostats | HTTP 401 | "Your Taostats API key is invalid or expired. Update it in Settings." |

| Wallet | Insufficient balance | "Your wallet balance ({balance} TAO) is less than your trading pot ({pot} TAO). Fund your wallet or reduce the pot size." |

#### "Refresh" Button

A manual refresh button in the bottom-right of the health card. Clicking it:
1. Shows a spinner on the button.
2. Calls `GET /api/health/services`.
3. Updates the display.
4. Button returns to normal.

The automatic refresh interval is 30 seconds (separate from the existing 10-second
control status poll).

#### Collapsed State

On mobile (viewport < 768px), the health card defaults to collapsed — showing only:
```
System Health: 4/5 services healthy  [▼]
```

Tapping expands to the full view. On desktop it is always expanded.

### 2.4 Frontend: Implementation Details

#### New TypeScript Types

```tsx
interface ServiceHealth {
  ok: boolean;
  name: string;
  detail: string;
  last_check: string;
  latency_ms?: number;
  // Wallet-specific
  can_trade?: boolean;
  balance_tao?: number;
  pot_tao?: number;
  // Optional services
  optional?: boolean;
}

interface HealthResponse {
  timestamp: string;
  services: Record<string, ServiceHealth>;
}
```

#### SWR Hook

```tsx
const { data: health, mutate: refreshHealth } = useSWR<HealthResponse>(
  `${API}/api/health/services`,
  fetcher,
  { refreshInterval: 30000 },
);
```

#### Health Card Component

Extract into a reusable `<HealthCard />` component within the control page (not a
separate file — the component is only used here):

```tsx
function HealthCard({ health, onRefresh }: {
  health: HealthResponse | undefined;
  onRefresh: () => void;
}) {
  // ... renders the service list, handles expand/collapse for errors
}
```

## Error Handling

| Scenario | Behaviour |
|----------|-----------|
| Health endpoint times out (>30s) | Frontend shows "Health check timed out" in the card with a retry button. |
| One service check fails but others succeed | Display partial results. Failed service shows its error. |
| Backend unreachable | Health card shows "Cannot reach backend" with existing error styling. The rest of the control page continues working from its own SWR hook (10s poll to `/api/control/status`). |
| Wallet balance check slow | The wallet row shows a spinner for up to 20s, then "Timed out". Other services display normally. |
| Go-live endpoint called with missing fields | Returns checks with `wallet_configured.ok: false`. Frontend keeps the "Enable Live Trading" button disabled. |

## Interaction with Existing Features

### Kill Switch

The health dashboard is informational only — it does NOT trigger the kill switch.
Even if all services are unhealthy, the bot continues running (or paused, if the kill
switch is already active). The dashboard helps the user diagnose *why* the bot isn't
trading, but doesn't take automated action.

### Setup Wizard

The safety gate modal in Step 4 reuses the same backend checks as the health dashboard
(`POST /api/config/go-live`) but presents them in a confirmation context. The wizard
does NOT show the full health dashboard — that's only on `/control`.

### Telegram `/status` Command

The existing `/status` Telegram command does NOT include service health details. It
continues to show portfolio status only. Service health is a visual dashboard concern —
operators troubleshoot from the browser, not from Telegram.

## Out of Scope

- Automated recovery actions (e.g., auto-restart on service failure) — operator decides.
- System resource monitoring (CPU, RAM, disk) — use standard OS tools.
- Log aggregation / rotation — handled by the logging module and `watchdog.sh`.
- API rate limit display for Taostats (30 req/min is fixed; no dynamic tracking).
- WebSocket-based real-time health updates (polling at 30s is sufficient).
- Health check history / uptime tracking — only current status is shown.
- Authentication on health endpoints — matches existing security model (localhost only).
